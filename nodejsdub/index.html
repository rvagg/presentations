LevelDB

"The Node.js of Databases"—@dominictarr

Presentation by Rod Vagg / tw:@rvagg / gh:rvagg

What?

  • An open-source, dependency-free, embedded key/value datastore
  • Developed in 2011 by Jeff Dean and Sanjay Ghemawat from Google
  • Based on ideas in Google's BigTable
  • Initially used as the store behind Chrome's IndexedDB

Features

  • Keys and values are arbitrary strings of bytes
  • Sorted by keys (pluggable comparator function)
  • Values are compressed with Snappy
  • Basic perations: Get(), Put(), Del(), Batch()
  • Iterators start & end anywhere, bi-directional
  • Snapshots

Negatives?

  • Single process access only (threads OK)
  • Simple key / value store, basic operations, no SQL
  • Heavy reliance on OS filesystem interface for performance

These are only negatives when viewed from a traditional "database" perspective

Basic architecture

LSM-tree

  • Writes (put & delete) go straight into a log, in memory and on disk
  • Log reaches size limit and is flushed in to multiple sorted table files
  • Reads involve a merging of the log and the table files to discover current entries
  • Configurable LRU cache stores keys & values to speed up common lookups

Basic architecture

Sorted table files (sst)

  • Limited to ~2Mb each by default
  • Divided into independently readable 4K blocks
  • Multiple entries per block & multiple blocks per entry
  • Index at the footer of each table file
  • Index references restarts, positioned every 16 entries
  • Bloom-filter used for best guess restart for lookups
  • Keys can have shared prefixes within blocks

Table file hierarchy

The "Level" in LevelDB

Log: Max size of 4Mb then flushed into a set of Level 0 sst files
Level 0: Max of 4 sst files then one file compacted into Level 1
Level 1: Max total size of 10Mb then one file compacted into Level 2
Level 2: Max total size of 100Mb then one file compacted into Level 3
Level 3+: Max total size of 10 x previous level then one file compacted into next level

0 ↠ 4 sst, 1 ↠ 10M, 2 ↠ 100M, 3 ↠ 1G, 4 ↠ 10G, 5 ↠ 100G, 6 ↠ 1T, 7 ↠ 10T

Batches, iterators & snapshots

Batch writes, consist of a list of Put and/or Del operations and may be considered to be atomic.

An Iterator provides a way to navigate forward and backward through the store, starting and ending at any key.

A Snapshot creates a consistent view of the whole data store as of the time it is created. Iterators create an implicit snapshot, active during their lifetime.

LevelUP & LevelDOWN

LevelDOWN is 99% C++ code, providing an interface layer between Node.js/V8/libuv and LevelDB. Exposes basic LevelDB operations with limited sugar. Completely async.

LevelUP wraps LevelDOWN to provide a Node.js-style interface to LevelDB.

  • Sugar: optional arguments, deferred-till-open operations
  • Safety: less foot-gun than LevelDOWN
  • Streams! Iterators are exposed as a ReadStream, a WriteStream can wrap Put and Batch operations
  • JSON & other encoding types

LevelUP: Simple example

var leveldb = require('levelup')

leveldb('/tmp/dprk.db', function (err, db) {
  db.put('name', 'Kim Jong-un', function (err) {
    db.batch([
        { type: 'put', key: 'spouse', value: 'Ri Sol-ju' }
      , { type: 'put', key: 'dob', value: '8 January 1983' }
      , { type: 'put', key: 'occupation', value: 'Clown' }
    ], function (err) {
      db.createReadStream()
        .on('data', console.log)
        .on('close', function () {
          db.close()
        })
    })
  })
})

LevelUP: basic operations

create / open / close

leveldb('/path/to/database', function (err, db) { /* use `db` */ })
// or
var db = leveldb('/path/to/database')
// use `db`, operations are queued till `open` is complete

db.close(function (err) { /* ... */ })

put / delete / get

db.put('key', 'value', function (err) { /* ... */ })

db.del('key', function (err) { /* ... */ })

db.get('key', function (err, value) { /* ... */ })

batch

var operations = [
    { type: 'put', key: 'Franciscus', value: 'Jorge Mario Bergoglio' }
  , { type: 'del', key: 'Benedictus XVI' }
]
db.batch(operations, function (err) { /* ... */ })

LevelUP: Streams

var rs = db.createReadStream()
rs.on('error', function (err) { /* handle err */ })
rs.on('data' , function (data) { /* data.key & data.value */ })
rs.on('close', function () { /* stream finished & cleaned up */ })

Options! Oh my!

db.createReadStream({
    start     : 'somewheretostart'
  , end       : 'endkey'
  , limit     : 100
  , reverse   : true
  , keys      : true // see db.createKeyStream()
  , values    : true // see db.createValueStream()
  , fillCache : false
})

Standard readable stream operations are also supported: pause(), resume(), destroy() and pipe()

function copy (srcdb, destdb, callback) {
  srcdb.createReadStream()
    .pipe(destdb.createWriteStream())
    .on('error', callback)
    .on('close', callback)
}

Encoding

var db = levelup('/path/to/db', { valueEncoding: 'json' })

db.put(
    'dprk'
  , {
        name       : 'Kim Jong-un'
      , spouse     : 'Ri Sol-ju'
      , dob        : '8 January 1983'
      , occupation : 'Clown'
    }
  , function (err) {
      db.get('dprk', function (err, value) {
        console.log('dprk:', value)
        db.close()
      })
    }
)

Use keyEncoding and valueEncoding to support utf8 Strings (default), JSON serialised objects and Buffers with different encodings.
